# LIFE OS - Product Requirements Document (PRD)
## 기준으로 돌아오는 루틴 앱

**버전**: v0.1  
**작성일**: 2025-01-27  
**상태**: MVP 설계

---

## 0. 이 앱을 만드는 이유

이 앱은 목표를 달성하기 위한 앱이 아니다.  
이 앱은 **무너져도 다시 기준으로 돌아오기 위한 앱**이다.

- 연속성보다 복귀
- 성과보다 기준
- 관리보다 회복

이 앱은 "잘 사는 사람"을 만들지 않는다.  
**"다시 사는 사람"을 만든다.**

---

## 1. 제품 개요

### 1.1 제품 비전
LIFE OS는 목표 달성을 위한 앱이 아닌, **무너져도 다시 기준으로 돌아오기 위한 앱**입니다. 연속성보다 복귀, 성과보다 기준, 관리보다 회복에 중점을 둡니다.

### 1.2 핵심 가치 제안
- **"다시 사는 사람"을 만드는 앱**: 완벽한 성취가 아닌, 무너진 후 복귀를 지원
- **Baseline 중심**: 3개의 절대 기준만으로 단순하고 명확한 구조
- **실패 개념 부재**: Reset은 실패가 아닌 복귀 선언
- **조용한 앱**: 알림 폭탄, 통계 과시, 경쟁 요소 제거

### 1.3 타겟 사용자
- 일상 루틴이 무너졌을 때 다시 돌아오고 싶은 사람
- 연속일과 점수에 압박받지 않고 싶은 사람
- 자신만의 기준을 지키고 싶은 사람

---

## 2. 핵심 철학 (Non-negotiable)

1. **Baseline이 전부다**: 3개의 절대 기준만 존재
2. **확장은 옵션이고, 복귀는 필수다**: 성장보다 회복 우선
3. **기록은 평가가 아니라 좌표다**: 데이터는 판단 도구가 아닌 위치 확인
4. **실패라는 개념은 존재하지 않는다**: Reset은 복귀 선언
5. **오늘의 성공은 '기준으로 돌아왔는가'다**: 달성률이 아닌 복귀 여부

---

## 3. 핵심 개념 정의

### 3.1 Baseline (절대 기준)

사용자는 딱 **3개의 절대 기준**만 가진다.

예시:
- 수면: 22:00–05:00
- 이동: 걷기/러닝 1km 이상
- 기록: 3줄

특징:
- 점수 없음
- 연속일 없음
- 달성 여부는 ON / OFF 뿐

### 3.2 Phase System (상태 기반 구조)

Phase는 성장 단계가 아니라 **현재 상태**를 나타낸다.

- Phase 1: Baseline - 기본 기준 회복
- Phase 2: Stability - 안정화 단계
- Phase 3: Growth - 성장 단계
- Phase 4: Identity - 정체성 확립

원칙:
- Phase는 자동으로 상승하지 않는다
- 사용자가 직접 선택한다
- 무너졌다고 Phase가 내려가지 않는다
- "나는 지금 Phase 1로 돌아간다"는 선택을 존중한다

### 3.3 Reset 개념 (이 앱의 차별점)

Reset은 실패가 아니다.  
Reset은 **복귀 선언**이다.

Reset 버튼을 누르면:
- 오늘의 목표는 Baseline 3개만 남는다
- 나머지 모든 목표/확장 기능은 숨겨진다
- 메시지:  
  > "오늘은 돌아오기만 하면 된다"

## 4. 기능 요구사항

### 4.1 MVP 화면 구성 (3개 화면만)

#### 4.1.1 Home 화면
**목적**: 오늘의 Baseline 확인 및 Reset 기능

**구성 요소**:
- 오늘 날짜 표시
- Baseline 3개 항목 (ON/OFF 토글)
  - 수면: 시간 범위 표시 (예: 22:00-05:00)
  - 이동: 거리/시간 표시 (예: 걷기/러닝 1km 이상)
  - 기록: 텍스트 표시 (예: 3줄)
- 🔄 Reset Today 버튼
- 현재 Phase 표시 (Phase 1-4)

**금지 요소**:
- 점수/점수화
- 연속일 카운터
- 랭킹/리더보드
- 통계 그래프
- 달성률 표시

**인터랙션**:
- Baseline 항목 클릭 시 ON/OFF 토글
- Reset Today 버튼 클릭 시:
  - Baseline 3개만 남고 나머지 숨김
  - 메시지 표시: "오늘은 돌아오기만 하면 된다"

#### 4.1.2 Daily Log 화면
**목적**: 하루의 기록과 상태 저장

**입력 항목**:
- 오늘의 기준 한 줄 (필수, 텍스트 입력)
- 몸 상태 (라디오 버튼: 좋음 / 보통 / 무거움)
- 자유 메모 (선택, 텍스트 영역)

**특징**:
- 빈 기록 저장 가능
- Baseline 체크를 못 해도 저장 가능
- "안 썼다"는 실패 개념 없음
- 저장 버튼: "저장" 또는 "기록하기"

#### 4.1.3 Phase 화면
**목적**: 현재 상태 선언 및 Phase 변경

**표시 내용**:
- 현재 Phase (Phase 1-4)
- Phase 설명 문구:
  - Phase 1: Baseline - 기본 기준 회복
  - Phase 2: Stability - 안정화 단계
  - Phase 3: Growth - 성장 단계
  - Phase 4: Identity - 정체성 확립

**기능**:
- Phase 수동 변경 (드롭다운 또는 버튼)
- "지금 나는 이 상태다"를 선언하는 공간
- Phase는 자동으로 상승하지 않음
- 무너졌다고 Phase가 자동으로 내려가지 않음

## 5. 데이터 모델

### 5.1 User
```typescript
{
  id: string (UUID)
  created_at: Date
  current_phase: number (1-4)
}
```

### 5.2 Baseline
```typescript
{
  sleep: string (예: "22:00-05:00")
  movement: number (예: 1, 단위: km)
  record: string (예: "3줄")
}
```

### 5.3 DailyLog
```typescript
{
  date: Date (YYYY-MM-DD)
  baseline_check: {
    sleep: boolean
    movement: boolean
    record: boolean
  }
  one_line: string (필수)
  body_state: "good" | "normal" | "heavy"
  memo: string (선택)
}
```

## 6. 기술 요구사항

### 6.1 기술 스택

#### 6.1.1 프론트엔드/백엔드
- **Next.js**: 풀스택 프레임워크로 프론트엔드와 백엔드 API 라우트 구현

#### 6.1.2 데이터베이스/스토리지/인증
- **Supabase**: 
  - PostgreSQL 데이터베이스 (User, Baseline, DailyLog 저장)
  - 실시간 데이터 동기화
  - 사용자 인증 (선택적, 로그인 없이도 사용 가능)
  - 스토리지 (필요 시 파일 저장)

#### 6.1.3 AI 기능
- **Gemini API**: AI 기반 기능 연동 (향후 확장 시 활용)

### 6.2 Local-first 원칙
- 모든 기록은 로컬 저장 우선 (Next.js 클라이언트 측 저장소)
- Supabase는 백업 및 동기화 역할
- 로그인 없이도 사용 가능 (익명 사용자 지원)
- 데이터 소유권은 사용자에게 있음

> 내 삶의 기준은 서버 장애와 함께 사라지지 않는다.

### 6.3 성능 요구사항
- 오프라인에서도 동작 (로컬 저장 우선)
- 빠른 로딩 속도 (< 2초)
- 데이터 동기화는 백그라운드에서만
- Next.js SSR/SSG 활용으로 초기 로딩 최적화

## 7. UI/UX 원칙

### 7.1 디자인 철학
- **최소주의**: 불필요한 요소 제거
- **조용함**: 알림 최소화, 침묵의 공간
- **친근함**: 압박감 없는 인터페이스
- **명확함**: 혼란 없는 단순한 구조

### 7.2 색상 가이드라인
- 부드러운 톤 사용 (과도한 빨강/경고색 지양)
- ON/OFF 상태는 명확하되 부드럽게
- Reset 버튼은 눈에 띄되 위협적이지 않게

### 7.3 타이포그래피
- 가독성 우선
- 친근한 폰트 사용
- 적절한 여백

## 8. MVP 범위 (절대 지킬 것)

### 8.1 포함되는 기능
- ✅ Home 화면 (Baseline 3개, Reset 버튼)
- ✅ Daily Log 화면 (기록 입력)
- ✅ Phase 화면 (Phase 변경)
- ✅ 로컬 저장소 연동 (Next.js 클라이언트 측)
- ✅ Supabase 연동 (백업 및 동기화)

### 8.2 MVP에서 제외되는 기능
- ❌ 통계 그래프
- ❌ 목표 관리 시스템
- ❌ 알림 시스템 (또는 최소한만)
- ❌ SNS 기능
- ❌ 연속일 카운터
- ❌ 점수/랭킹 시스템
- ❌ Gemini API 연동 (MVP 이후)

이 앱은 **조용해야 한다**.

## 10. 향후 로드맵

### 10.1 개인 고도화
- 주간 회고 기능
- Baseline 변경 히스토리
- "이번 주 나는 몇 번 돌아왔는가" 통계
- Gemini API를 활용한 일기 분석 및 인사이트 제공

### 10.2 서비스화 가능 지점
- Phase 상태 공유 (숫자 없음)
- "나는 지금 회복 중입니다" 같은 상태 메시지
- Gemini API를 활용한 개인화된 조언 및 회복 지원

### 10.3 철학 유지 조건
- ⚠️ 연속일 기능 추가 시 → 앱 정체성 붕괴
- ⚠️ 점수화 시 → 이 앱은 폐기 대상
- ⚠️ AI 기능이 압박감을 주는 방향으로 사용될 시 → 철학 훼손

## 11. 리스크 및 제약사항

### 11.1 기술적 리스크
- 로컬 저장소 용량 제한
- 브라우저 호환성
- Supabase 데이터 백업 실패
- Next.js 빌드 및 배포 이슈
- Gemini API 사용량 및 비용 관리

### 11.2 제품 리스크
- 기능 추가 욕구로 인한 복잡도 증가
- 사용자 요구로 인한 철학 훼손
- 경쟁 앱과의 차별화 유지
- AI 기능이 압박감을 주는 방향으로 발전할 위험

### 11.3 완화 방안
- Baseline 문서를 정기적으로 재검토
- MVP 범위를 엄격히 준수
- 사용자 피드백보다 철학 우선
- AI 기능은 조용한 지원 역할로만 활용

## 12. 개발 중 나에게 남기는 경고

- 기능을 늘리고 싶어질 때 → Baseline 문서를 다시 읽어라
- 남에게 보여주고 싶어질 때 → 이 앱은 먼저 나를 위한 것이다
- 성과가 보이고 싶어질 때 → 성과는 삶이 증명한다
- AI 기능을 추가하고 싶어질 때 → 이 앱은 조용해야 한다
