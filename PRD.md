# LIFE OS - Product Requirements Document (PRD)
## 기준으로 돌아오는 루틴 앱

**버전**: v0.1  
**작성일**: 2025-01-27  
**상태**: MVP 설계

---

## 1. 제품 개요

### 1.1 제품 비전
LIFE OS는 목표 달성을 위한 앱이 아닌, **무너져도 다시 기준으로 돌아오기 위한 앱**입니다. 연속성보다 복귀, 성과보다 기준, 관리보다 회복에 중점을 둡니다.

### 1.2 핵심 가치 제안
- **"다시 사는 사람"을 만드는 앱**: 완벽한 성취가 아닌, 무너진 후 복귀를 지원
- **Baseline 중심**: 3개의 절대 기준만으로 단순하고 명확한 구조
- **실패 개념 부재**: Reset은 실패가 아닌 복귀 선언
- **조용한 앱**: 알림 폭탄, 통계 과시, 경쟁 요소 제거

### 1.3 타겟 사용자
- 일상 루틴이 무너졌을 때 다시 돌아오고 싶은 사람
- 연속일과 점수에 압박받지 않고 싶은 사람
- 자신만의 기준을 지키고 싶은 사람

---

## 2. 핵심 철학 (Non-negotiable)

1. **Baseline이 전부다**: 3개의 절대 기준만 존재
2. **확장은 옵션이고, 복귀는 필수다**: 성장보다 회복 우선
3. **기록은 평가가 아니라 좌표다**: 데이터는 판단 도구가 아닌 위치 확인
4. **실패라는 개념은 존재하지 않는다**: Reset은 복귀 선언
5. **오늘의 성공은 '기준으로 돌아왔는가'다**: 달성률이 아닌 복귀 여부

## 3. 기능 요구사항

### 3.1 MVP 화면 구성 (3개 화면만)

#### 3.1.1 Home 화면
**목적**: 오늘의 Baseline 확인 및 Reset 기능

**구성 요소**:
- 오늘 날짜 표시
- Baseline 3개 항목 (ON/OFF 토글)
  - 수면: 시간 범위 표시 (예: 22:00-05:00)
  - 이동: 거리/시간 표시 (예: 걷기/러닝 1km 이상)
  - 기록: 텍스트 표시 (예: 3줄)
- 🔄 Reset Today 버튼
- 현재 Phase 표시 (Phase 1-4)

**금지 요소**:
- 점수/점수화
- 연속일 카운터
- 랭킹/리더보드
- 통계 그래프
- 달성률 표시

**인터랙션**:
- Baseline 항목 클릭 시 ON/OFF 토글
- Reset Today 버튼 클릭 시:
  - Baseline 3개만 남고 나머지 숨김
  - 메시지 표시: "오늘은 돌아오기만 하면 된다"

#### 3.1.2 Daily Log 화면
**목적**: 하루의 기록과 상태 저장

**입력 항목**:
- 오늘의 기준 한 줄 (필수, 텍스트 입력)
- 몸 상태 (라디오 버튼: 좋음 / 보통 / 무거움)
- 자유 메모 (선택, 텍스트 영역)

**특징**:
- 빈 기록 저장 가능
- Baseline 체크를 못 해도 저장 가능
- "안 썼다"는 실패 개념 없음
- 저장 버튼: "저장" 또는 "기록하기"

#### 3.1.3 Phase 화면
**목적**: 현재 상태 선언 및 Phase 변경

**표시 내용**:
- 현재 Phase (Phase 1-4)
- Phase 설명 문구:
  - Phase 1: Baseline - 기본 기준 회복
  - Phase 2: Stability - 안정화 단계
  - Phase 3: Growth - 성장 단계
  - Phase 4: Identity - 정체성 확립

**기능**:
- Phase 수동 변경 (드롭다운 또는 버튼)
- "지금 나는 이 상태다"를 선언하는 공간
- Phase는 자동으로 상승하지 않음
- 무너졌다고 Phase가 자동으로 내려가지 않음

## 4. 데이터 모델

### 4.1 User
```typescript
{
  id: string (UUID)
  created_at: Date
  current_phase: number (1-4)
}
```

### 4.2 Baseline
```typescript
{
  sleep: string (예: "22:00-05:00")
  movement: number (예: 1, 단위: km)
  record: string (예: "3줄")
}
```

### 4.3 DailyLog
```typescript
{
  date: Date (YYYY-MM-DD)
  baseline_check: {
    sleep: boolean
    movement: boolean
    record: boolean
  }
  one_line: string (필수)
  body_state: "good" | "normal" | "heavy"
  memo: string (선택)
}
```

## 5. 기술 요구사항

### 5.1 Local-first 원칙
- 모든 기록은 로컬 저장 우선 (LocalStorage/IndexedDB)
- 서버는 백업 역할만 수행
- 로그인 없이도 사용 가능
- 데이터 소유권은 사용자에게 있음

### 5.2 기술 스택 (제안)
- **프론트엔드**: React/Next.js 또는 Vue/Nuxt.js
- **로컬 저장소**: LocalStorage 또는 IndexedDB
- **백업 서버**: Firebase/Supabase (옵션)
- **배포**: GitHub Pages 또는 Vercel

### 5.3 성능 요구사항
- 오프라인에서도 동작
- 빠른 로딩 속도 (< 2초)
- 데이터 동기화는 백그라운드에서만

## 6. UI/UX 원칙

### 6.1 디자인 철학
- **최소주의**: 불필요한 요소 제거
- **조용함**: 알림 최소화, 침묵의 공간
- **친근함**: 압박감 없는 인터페이스
- **명확함**: 혼란 없는 단순한 구조

### 6.2 색상 가이드라인
- 부드러운 톤 사용 (과도한 빨강/경고색 지양)
- ON/OFF 상태는 명확하되 부드럽게
- Reset 버튼은 눈에 띄되 위협적이지 않게

### 6.3 타이포그래피
- 가독성 우선
- 친근한 폰트 사용
- 적절한 여백

## 7. MVP 범위 (절대 지킬 것)

### 7.1 포함되는 기능
- ✅ Home 화면 (Baseline 3개, Reset 버튼)
- ✅ Daily Log 화면 (기록 입력)
- ✅ Phase 화면 (Phase 변경)
- ✅ 로컬 저장소 연동

### 7.2 MVP에서 제외되는 기능

- ❌ SNS 기능
- ❌ 연속일 카운터
- ❌ 점수/랭킹 시스템
- ❌ 서버 동기화 (MVP 이후)

## 8. 사용자 시나리오

### 시나리오 1: 하루 시작
1. 앱 실행 → Home 화면
2. 오늘 날짜 확인
3. Baseline 3개 항목 확인
4. 필요 시 Reset Today 버튼 클릭
5. Baseline 항목 달성 시 ON으로 토글

### 시나리오 2: 하루 기록
1. Daily Log 화면 이동
2. 오늘의 기준 한 줄 입력
3. 몸 상태 선택
4. 자유 메모 입력 (선택)
5. 저장

### 시나리오 3: Phase 변경
1. Phase 화면 이동
2. 현재 Phase 확인
3. 필요 시 Phase 변경
4. 변경 사유는 선택 사항

### 시나리오 4: 무너진 날 Reset
1. Home 화면에서 Reset Today 버튼 클릭
2. "오늘은 돌아오기만 하면 된다" 메시지 확인
3. Baseline 3개만 남고 나머지 숨김
4. Baseline 달성에 집중

## 9. 성공 지표 (MVP 이후)

### 정성적 지표
- 사용자가 압박감을 느끼지 않는가?
- Reset 기능을 편안하게 사용하는가?
- Baseline이 명확하게 인식되는가?

### 정량적 지표 (선택적)
- 일일 활성 사용자 (DAU)
- Baseline 달성률 (개인용, 공개 아님)
- Reset 사용 빈도

## 10. 향후 로드맵

### 10.1 개인 고도화
- 주간 회고 기능
- Baseline 변경 히스토리
- "이번 주 나는 몇 번 돌아왔는가" 통계

- ⚠️ 연속일 기능 추가 시 → 앱 정체성 붕괴
- ⚠️ 점수화 시 → 이 앱은 폐기 대상

## 11. 리스크 및 제약사항
